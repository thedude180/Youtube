**What:** The AI slot is acquired and held without `releaseAISlot()` being called, for 8-9 minutes, repeatedly at near-perfect 9-minute intervals: 16:34 → 16:43 → 16:53 → 17:02 → 17:11. **Why it matters:** This blocks ALL AI-dependent work for the entire session after 16:33. Publishers need AI for SEO/title generation. Back-catalog engine needs it for queuing decisions. VOD-shorts-loop drops title optimizations. The 4/4 background queue fills instantly and stays full between watchdog cycles.

**Root cause (suspected):** Job 18102 (V_fIPnGxHRs, the 6h BF2042 stream) was reset to "queued" by stream-editor at 16:33. The stream editor processes it, acquires an AI slot to call Claude/OpenAI for stream analysis, but the API call hangs (very large context → very long completion). Watchdog kills after 8-9 min. Stream editor retries (re-queues → re-acquires slot → hangs again).

**Durable lesson:** Migration 087 (cancel job 18102) must be deployed immediately. Any stream-edit job longer than ~2h of content should be gated with an AI context size check before acquiring the slot. The 9-minute rhythm is diagnostic: it's exactly the slot watchdog timeout, meaning the same caller keeps acquiring and hanging.

- All major AI engines are at or near exhaustion by 10 AM Pacific every day

**Lesson:** Front-load all publishing and heavy AI work to 07:00–09:30 Pacific (immediately after midnight UTC quota reset at 00:00 UTC = 5:00 PM Pacific previous day... wait, Pacific is UTC-7 in summer → midnight UTC = 5 PM Pacific. So quota resets at 5 PM Pacific. Publishing window: 5 PM → midnight Pacific = 7 hours. Heavy work should run 17:00–23:00 Pacific (00:00–06:00 UTC), NOT during 09:00–17:00 UTC.

### Migration 086 UPDATE 0 rows pattern
Migration 086 ran `UPDATE back_catalog_videos SET is_shorts_mined=true WHERE youtube_video_id='hBylGNbIT88'` and got 0 rows (video not imported yet). Migration marked done anyway (correct). 

**Lesson:** Migrations that UPDATE specific video IDs in back_catalog_videos may get 0 rows if the video was never imported. This is always safe because: (a) `setFlag()` marks it done, (b) 0 rows isn't an error, (c) when the video IS later imported, the back-catalog engine will process it normally. Do NOT add `RETURNING` checks that would treat 0 rows as a migration failure.

### InnerTube ANDROID HTTP 400 — 16 new video IDs

Since the server is still running (no crash today), failCount may be accumulating and these will eventually self-heal. But a migration to permanently fail the 16 confirmed-400 video IDs would stop the retry storm immediately.

### Goal planner: long-form pipeline "severely behind"
At 16:54 UTC, goal-planner flagged the long-form pipeline as "severely behind" and urged prioritizing segment downloads and pre-encoder queue. Combined with the AI semaphore saturation, this means the pre-encoder can't work because the AI slots are all held by the stuck-slot problem.
